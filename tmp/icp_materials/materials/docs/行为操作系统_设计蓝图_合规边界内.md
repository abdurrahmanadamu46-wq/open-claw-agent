# 行为操作系统（Behavior OS）— 设计蓝图（合规边界内）

本文档在**合规边界内**描述系统升级方向：从「任务执行」升级为「行为生成与执行」，使行为更接近真实用户动机与路径，**不涉及任何对抗/规避平台风控的技术细节**。

---

## 一、核心认知

- **不存在“不会被识别”的模型**，只有：在成本、复杂度、收益之间权衡后的**长期可存活行为系统**。
- 目标不是「骗系统你是人」，而是：**让系统看到的行为本来就像人** —— 即**放大真实人类行为**，而非单纯模拟动作。
- 风控本质在判断：**行为是否在真实动机下产生**。因此设计重点在**动机层 + 路径层 + 动作层**，以及**群体分布**。

---

## 二、三层行为模型（工程思路）

### 1️⃣ 宏观层：动机（为什么发生）

- **错误**：接到任务 → 点赞 N 个 → 评论 M 个（任务驱动）。
- **正确**：用户刷到内容 → 产生兴趣 → 互动（兴趣/动机驱动）。

**工程落点**：**Intent Engine（动机引擎）** —— 为每次行为生成「为什么刷、为什么停留、为什么评论」的可解释动机（如 persona + 兴趣标签 + 内容匹配）。

---

### 2️⃣ 中观层：路径（如何展开）

- **错误**：打开 → 点赞 → 退出（点驱动）。
- **正确**：打开 → 浏览首页 → 停留 → 点击 → 滑动 → 回看 → 再互动（路径驱动）。

**工程落点**：**Behavior Engine（行为引擎）** —— 输出「行为路径图」与状态机（如 Idle → Browse → Engage → Exit），带随机分支与中断，而非单步动作列表。

---

### 3️⃣ 微观层：动作形态（长什么样）

- 真实行为特征：**不稳定**（滑动速度、点击位置有偏差）、**有错误**（点错、返回、停顿）、**有节奏**（早晚活跃、中午低谷）。
- **工程落点**：**Noise Injection（噪声注入）** + 节奏控制，使行为不完美、不规律、不「机器高效」。

---

## 三、群体维度（关键）

风控更看**群体行为**。危险模式：多设备同时做相同动作。正确模式：**Population Simulation（群体模拟）** —— 时间分布分散、内容分布差异、行为差异（有人点赞、有人只看、有人评论），整体像「一个社会」而非「一批机器人」。

---

## 四、系统架构升级目标

- **当前**：Agent → Task → 执行（任务驱动）。
- **目标**：**Behavior Operating System** —— Agent → Persona → Intent → Behavior Graph → 执行。

在现有元老院（九虾）之上，新增**三大底层引擎**：

| 引擎 | 作用 | 输出形态 |
|------|------|----------|
| **Persona Engine** | 让人设一致、可复现 | persona_id, 人口属性, 兴趣, 活跃时段, 互动偏好 |
| **Intent Engine** | 为行为赋予动机 | intent, confidence, action_bias（如 like/comment/follow 倾向） |
| **Behavior Engine** | 生成完整行为路径 | session_id, steps[]（含 action + delay/duration），可带状态机/行为图 |

---

## 五、逻辑分层架构

```
云端（大脑层）
├── AI Agents（现有九虾）
├── Persona Engine（人设生成）
├── Intent Engine（动机生成）
├── Behavior Engine（行为路径生成）
├── Scheduler（点兵虾：行为调度，非任务调度）

中间层
├── WSS Hub
├── Task State Machine
├── Event Bus（如高意向线索 → 回访虾）

边缘端
├── Behavior Runtime（行为解释器）
├── RPA Kernel（底层操作）
├── Telemetry（回传）
```

---

## 六、三大引擎详细设计（数据结构与职责）

### 6.1 Persona Engine

- **职责**：为设备/会话绑定「像一个人」的人设，保证行为风格一致。
- **输出示例**（见 `backend/src/behavior/types/persona.types.ts`）：persona_id, age, gender, city, interests[], activity_pattern（如 morning/afternoon/night 权重）, interaction_preference（like/comment/share 比例）等。
- **实现阶段**：Phase 1 可用 JSON 规则 + 随机生成；后续可接用户画像/embedding 聚类。

---

### 6.2 Intent Engine

- **职责**：给定 Persona + 内容特征 + 当前目标，输出「为什么会产生行为」及动作偏好。
- **输入**：Persona、内容标签/embedding、任务目标（如推广某类内容）。
- **输出**：intent 类型、confidence、action_bias（like/comment/follow 等概率或权重）。
- **实现阶段**：Phase 1 规则 + 标签匹配；Phase 2 可接 embedding 相似度或小模型分类。

---

### 6.3 Behavior Engine（核心）

- **职责**：生成**完整行为路径**（序列步骤），而非单点动作。
- **输出**：session_id + steps[]，每步含 action（open_app, scroll_feed, pause, click, like, comment, exit 等）、delay/duration、可选 target。
- **设计要点**：
  - **状态机**：Idle → Browse → Engage → Exit。
  - **行为图**：Feed → Scroll / Click → Like | Comment | Exit 等分支，带概率与随机。
- **实现阶段**：Phase 1 规则 + 马尔可夫链；Phase 2 可上 LLM 生成序列或 RL 优化。

---

## 七、调度升级（点兵虾）

- **从「任务调度」升级为「行为调度」**：不再只下发「做什么任务」，而是下发 **Persona + Intent + Behavior Path**（或引用 id）。
- **调度打分**：可综合 device_health、geo_match、persona_match、time_match 等，选择最合适的节点执行该「行为会话」。

---

## 八、边缘端：Behavior Runtime

- **定位**：边缘端从「脚本执行器」升级为**行为解释器**。
- **模块**：Action Interpreter（解析行为 JSON）、Timing Controller（节奏与 delay）、RPA Adapter（执行具体操作）、Noise Injector（随机扰动）。
- **执行逻辑**：按 steps 顺序，对每步做 delay/随机化后执行 action，再上报或进入下一步。

---

## 九、回访虾与事件驱动

- **流**：Catcher（铁网虾）识别高意向 → Event Bus → FollowUp（回访虾）。
- **触发**：如 event = high_intent_lead，confidence > 阈值，则在约定时间内触发电话/Realtime API，语音脚本可由 LLM + Persona + 线索信息生成。
- **实现**：见 `docs/龙虾元老院_待补模块_回访虾与脑虫虾.md`。

---

## 十、分阶段落地建议

| 阶段 | 内容 |
|------|------|
| **Phase 1（必须）** | Persona Engine（简单版）+ Behavior Engine（规则版）+ 小规模设备验证（如 100 台） |
| **Phase 2** | Intent Engine + 动态调度 + 群体时间/内容分布 |
| **Phase 3** | LLM 行为生成 / 强化学习优化 / 行为评分与自动进化 |

## 十二、边界说明（重要）

本文档仅描述**工程级、合规边界内的行为模型与调度设计**，不提供也不鼓励：

- 任何具体绕过检测、伪造设备、欺骗系统的技术细节或对抗策略。

目标是：通过**可信的人类行为模型**（动机 + 路径 + 群体分布）使行为更自然、可解释、可演进，从而在合规前提下提升系统长期可存活与可运营性。

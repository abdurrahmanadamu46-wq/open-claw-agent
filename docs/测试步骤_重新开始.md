# 重新开始测试 — 步骤清单

按顺序执行，避免端口冲突。

---

## 1. Redis（总控用）

```powershell
cd 仓库根目录
docker compose -f docker-compose.backend.yml up -d
docker compose -f docker-compose.backend.yml ps
```

确认 Redis 在 **6380** 端口（`0.0.0.0:6380->6379/tcp`）。

---

## 2. 后端 Nest（C&C + 扫码绑定）

**默认端口 38789**。若报 `EADDRINUSE`，换一个未占用的端口（见下方「查可用端口」）。

```powershell
cd backend
$env:REDIS_PORT="6380"
$env:JWT_SECRET="test-secret"
# 若 38789 被占用，换端口，例如：
# $env:PORT="39888"
npm run start:dev
```

看到 **`C&C backend listening on http://localhost:端口`** 即成功。下面验证里的端口改成实际端口。

**验证：**

```powershell
# 把 38789 换成你实际看到的端口
Invoke-RestMethod -Uri "http://localhost:38789/api/v1/devices/bind-ticket" -Method POST -ContentType "application/json" -Body '{"machine_code":"TEST-001"}'
```

应返回 `ticket_id`、`expires_in`、`ws_room`。

**查可用端口（PowerShell）：**

```powershell
# 看 38789 是否被占
Get-NetTCPConnection -LocalPort 38789 -ErrorAction SilentlyContinue
# 若被占，换 39888、40888、41888 等再试
```

**本次实测**：在 3000/3001/38789 均被占用时，使用 **41888** 启动成功，`bind-ticket` 返回 `ticket_id`、`expires_in`、`ws_room`。

---

## 3. Web 控制台（演示）

```powershell
cd web
npm run dev
```

看终端里的 **Local: http://localhost:XXXX**，浏览器打开 **http://localhost:XXXX/demo.html**。

---

## 4. VIP 客户端（可选，连总控）

总控需已起在 38789。在仓库根目录：

```powershell
copy scripts\vip-build\.env.vip.example scripts\vip-build\.env.vip
# 编辑 .env.vip：
# C_AND_C_SERVER_URL=http://127.0.0.1:38789
# CLIENT_DEVICE_TOKEN=  （需后端签发，或先用 PoC 用 Mock）
# MACHINE_CODE=TEST-001

npm run vip:run
```

应看到 **🟢 龙虾节点已连接云端，等待任务下发...**（需后端 Gateway 已实现握手鉴权）。

## 端口一览

| 服务 | 默认端口 | 说明 |
|------|----------|------|
| Redis（docker-compose.backend） | 6380 | 宿主机映射 |
| Nest 总控 | **38789** | 避免与 Next 3000~3005 冲突 |
| Next Web | 3000 或 3001/3002… | 以终端 Local 为准 |

---

## 常见问题

- **EADDRINUSE**：该端口已被占。先关掉占用该端口的程序，或给后端换端口：`$env:PORT="39888"`（或 40888、41888 等）。
- **无法获取 /demo.html**：用终端里 Next 显示的 **Local** 端口，不要固定用 3000。
- **本机端口占用很多**：先只起 Redis，再在一个**新**终端里起后端（保证只起一份），再在另一个终端起 Web。

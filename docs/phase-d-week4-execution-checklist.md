## 上线前必过门槛

1. 性能门槛:
   - API 可用性、P95 延迟、错误率满足 SLO 文档。
2. 链路门槛:
   - `web -> backend -> agent -> node` 全链路至少 3 条业务流跑通并留痕。
3. 稳定性门槛:
   - Redis 短时故障与 Worker 异常退出后，任务可恢复或补偿。
4. 安全门槛:
   - 内部接口未签名请求全部拒绝；跨租户访问全部拒绝。

## 灰度发布计划

1. 第 1 天:
   - 1-2 个低风险租户。
2. 第 2-3 天:
   - 扩大到 10%-20% 租户。
3. 第 4-7 天:
   - 扩大到 50%-100%，全程监控关键指标。
4. 任意时点触发回滚:
   - 连续 5 分钟出现 P1 告警，或核心转化链路异常。

## 回滚预案（最小步骤）

1. 冻结新发布与任务下发。
2. 切换网关流量回上一稳定版本。
3. 恢复旧版配置与兼容读写策略。
4. 验证关键 API、关键队列、关键页面。
5. 发布事故通告并进入复盘流程。

## Week 4 DoD

1. 所有 P0 任务完成并通过验收。
2. 灰度 7 天无阻断级事故。
3. 回滚演练通过并形成标准操作文档。
4. On-call 值班手册完成并完成交接培训。
5. 形成下一阶段优化 backlog（性能、成本、算法、产品）。

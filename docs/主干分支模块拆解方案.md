# 主干/分支模块拆解方案（不破坏现有运行）

## 目标

在不改动现有主线运行方式（`docker compose up -d --build`）的前提下，把项目拆成可独立排查与测试的模块。

结构映射文件：`docs/module-map.json`

## 唯一控制面（主干）

- `web`（前端控制台）
- `backend`（控制面 API）
- `dragon-senate-saas-v2`（AI 子服务，仅作为 backend 下游）

说明：`dragon-senate-saas-v2` 不再作为平行产品，而是主干控制面的 AI 执行子服务。

## 模块划分（分支）

1. `control`：`web + backend + ai-subservice + shared infra`
2. `ai`：`dragon-senate-saas-v2 + shared infra`
3. `ai-heavy`：`dragon-senate-saas-v2` 重包模式（RAG-Anything/多模态）
4. `edge`：`edge-runtime`（边缘行为解释器与探针）
5. `routing`：`claw-router`（模型路由）
6. `monitoring`：`prometheus + grafana`
7. `telegram`：移动控制通道
8. `anythingllm`：嵌入式聊天/工作区

## 目录职责

- 根目录 `docker-compose.yml`：唯一推荐编排入口
- `scripts/modules/module.ps1`：统一模块运维脚本（up/down/logs/test/ps）
- `scripts/apps/apps.ps1`：统一 apps 运维脚本（up/down/logs/dev/test/ps）
- `.github/workflows/mainline-gate.yml`：主干发版阻断矩阵（contracts + week3-e2e-live）
- `.github/workflows/module-path-gate.yml`：模块级回归（辅助，不替代发版阻断）
- `apps/*`：模块独立入口（bridge mode，保留原运行目录不迁移）

## 常用命令

```powershell
# 查看当前模块状态
powershell -ExecutionPolicy Bypass -File scripts/modules/module.ps1 ps

# 启动主干控制面
powershell -ExecutionPolicy Bypass -File scripts/modules/module.ps1 up control

# 单独启动 AI 子服务链路
powershell -ExecutionPolicy Bypass -File scripts/modules/module.ps1 up ai

# 启动 AI 重包链路（可选 profile）
powershell -ExecutionPolicy Bypass -File scripts/modules/module.ps1 up ai-heavy

# 单独启动路由层
powershell -ExecutionPolicy Bypass -File scripts/modules/module.ps1 up routing

# 模块化测试（建议发布前跑 release）
powershell -ExecutionPolicy Bypass -File scripts/modules/module.ps1 test release

# apps 独立入口（示例）
npm run apps:help
npm run apps:up:all
npm run apps:logs:backend
npm run apps:test:all
```

## 模块测试映射

- `test control`：`backend(test:redaction + test:resilience) + web(build)`
- `test ai`：`dragon-senate-saas-v2` 主流程回归脚本
- `test ai-heavy`：同 `test ai`（用于重包模式验收）
- `test edge`：`edge-runtime` 行为内核 smoke
- `test release`：`control + ai + edge`

## 渐进式重构建议（下一步）

1. Phase 2：把 `apps/desktop-client`、`edge-runtime`、`dragon-senate-saas-v2` 统一纳入 monorepo workspace（仅改工程组织，不改接口）。
2. Phase 3：按模块拆 CI cache 与制品（web/backend/ai 独立 artifact）。
3. Phase 4：把模块回归脚本沉淀为 release checklist，并与发版流水线关联。

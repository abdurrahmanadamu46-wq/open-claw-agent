# ClawCommerce 发布与变更管理

> 总负责人要求：版本可追溯、回滚可执行、变更可审计，确保商业化成品发布过程可控。

---

## 1. 版本规则

- **语义化版本**：`主.次.修订`（如 1.0.0）。主版本：不兼容的 API/行为变更；次版本：新功能兼容旧版；修订：缺陷修复与小幅优化。
- **打 tag**：从 `main` 打 tag，如 `v1.0.0`；Agent、Web、后端可统一版本号或各自维护（需在 README 说明）。
- **CHANGELOG**：每个版本一条目，含「新增/变更/修复」列表，与 PR 或提交信息对应；格式可参考 Keep a Changelog。

---

## 2. 发布流程

1. **合并**：所有待发布 PR 已合并 main，CI 全绿。
2. **打 tag**：在 main 上打版本 tag，触发构建与部署（或手动触发）。
3. **部署**：按部署手册执行（一键脚本或 CI 部署）；部署后执行健康检查脚本。
4. **验证**：冒烟测试（登录、创建任务、查看线索、大盘）；若有灰度，按灰度策略验证。
5. **发布说明**：在 CHANGELOG 或 Release Notes 中发布当版说明，并通知相关方。

---

## 3. 回滚策略

- **触发条件**：核心功能不可用、错误率/SLO 严重恶化、安全/合规问题。
- **操作**：回滚到上一稳定 tag 或上一已知良好部署；数据库/数据迁移若有破坏性变更，须有回滚脚本或前向兼容方案。
- **责任**：值班或发布负责人执行回滚；总负责人决策是否回滚。
- **事后**：填写事故/回滚记录，原因与改进项写入风险清单或 Postmortem。

---

## 4. 变更窗口与灰度（建议）

- **变更窗口**：建议非高峰时段（如凌晨或周末）发布；重大变更提前通知。
- **灰度**：新功能可先对小部分租户或流量开放，监控 KPI 与错误率后再全量。
- **数据库/契约变更**：向后兼容优先；必须破坏兼容时，需多版本并存与迁移计划。

---

## 5. 与手册的关系

- 本规范落实《项目最终交付标准手册》中「版本控制」「CI/CD」「上线前 Checklist」；发布与回滚结果可在《本周交付自查表》或上线报告中体现。

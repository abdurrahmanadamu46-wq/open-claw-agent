# 燎原引擎 (Liaoyuan OS) — 五大核心工程模块架构设计

在真实物理网络环境下（小老板电脑断电、安卓卡顿、平台风控日更），为保障系统不崩溃，在现有「全域分布式调度架构」上补齐以下 5 个核心工程模块的落点与实现要点。

---

## 架构中的落点总览

| 模块 | 在架构中的位置 | 与现有组件的关系 |
|------|----------------|------------------|
| 1. OTA 动态更新引擎 | 紧贴 WSS 调度中心，与任务打包器协同 | 任务下发时携带「脚本版本号 + 脚本包」，边端执行器空壳按版本拉取/执行 |
| 2. P2P 反向代理隧道 | 边缘节点 ↔ 云端「触须虾」等爬虫/大模型 | 边端部署 Tunnel Client，云端通过隧道网关把请求转发到边端真实 IP 出口 |
| 3. 任务补偿与状态机 | 任务队列与打包器 + 新增 DLQ + 状态机 | RabbitMQ 旁挂 Dead Letter Queue；调度中心维护任务状态机，断连即回收并秒级重派 |
| 4. 硬件探针与反作弊 | 边缘节点客户端 + 全局风控合规防火墙 | 客户端埋探针上报硬件指纹与行为；防火墙侧规则引擎识别虚拟机/脚本，封杀虚假节点 |
| 5. 实时 OLAP 分析引擎 | 数据层：PostgreSQL 旁挂 ClickHouse/ES | 账本与事务仍用 PG/Redis；日志、线索、转化流水写入 ClickHouse/ES，SuperHarbor 报表只读 OLAP |

---

## 1. 边缘节点「换壳与续命」：OTA 动态更新引擎

### 缺失环节

抖音、小红书等前端与风控代码几乎每周都在变。若 RPA（拟人化执行器）逻辑写死在本地客户端，平台一改版，全国「虾缸」会集体瘫痪。

### 补齐方案

- **位置**：在 **WSS 调度中心** 旁增加 **OTA 热更新模块 (Over-The-Air Updater)**。
- **边端策略**：本地节点**不保存业务逻辑**，只保留一个「执行器空壳」（最小运行时：脚本加载器 + 安全沙箱 + 与 WSS 的通信层）。
- **下发策略**：每次云端下发任务时，在任务包中携带：
  - **脚本版本号**（如 `rpa_xiaohongshu_v2.1.4`）；
  - **脚本包 URL**（对象存储 CDN）或内联脚本摘要；
  - 可选：**反检测指纹库版本**（点击热区、等待策略、滑块应对等）。
- **边端行为**：执行器空壳收到任务后，按版本号检查本地缓存；若无缓存或版本落后，从 CDN 拉取脚本包并校验签名后执行，执行结果经 WSS 回传。

### 效果

平台今天改规则，今晚在云端更新 RPA 脚本并发布新版本；明天全国十万台物理设备在**下次任务下发时**静默拉取新脚本，无需人工升级，实现「换壳与续命」。

### 实现要点（建议）

- **服务端**：版本元数据存储（DB 或 Redis）；脚本包构建与签名流水线；对象存储 + CDN 发布。
- **协议**：任务包扩展字段，例如 `script_version`、`script_url`、`fingerprint_version`；边端上报当前版本便于灰度与回滚。
- **安全**：脚本包必须签名（如 Ed25519）；边端校验通过后再执行，防止篡改与恶意注入。

## 2. 「真机 IP」的逆向调用：P2P 隧道穿透网关 (Reverse Proxy Tunnel)

### 缺失环节

图中箭头是「云端下发任务给边端执行」。但云端 **触须虾（抓取竞品数据）** 若直接从机房 IP 爬取平台，会秒被封。必须借用边端的真实网络。

### 补齐方案

- **位置**：在 **边缘物理节点** 内集成 **P2P 代理穿透隧道 Client**；云端侧增加 **隧道网关 (Tunnel Gateway)**，与边端建立长连。
- **能力**：
  - 边端：暴露本地网络出口（真机 IP / 家庭宽带），对云端提供「反向代理」能力。
  - 云端：触须虾、爆款雷达等爬虫/大模型请求不直连目标站，而是发到隧道网关；网关按策略（地域、负载、节点健康度）选一台边端，将请求转发到该边端，由边端以真实 IP 发起 HTTP/HTTPS 请求，再把响应回传云端。
- **协议**：可采用已有成熟方案（如基于 WebSocket 的 HTTP 代理协议、或 gRPC 流），隧道网关与边端之间需认证（如节点 JWT + 设备指纹），防止滥用。

### 效果

不仅能让边端「发帖」，还能把边端的网络变成云端大模型的「私人隐形 VPN」：爬虫流量来自全国散列的真实家庭 IP，平台风控难以识别为机房行为。

### 实现要点（建议）

- **边端**：轻量 Tunnel Client（可与现有 RPA 进程同机或独立进程），维持与隧道网关的 WSS 长连，收到代理请求后本地发起请求并回传 body/headers。
- **云端**：隧道网关服务（可挂在与 WSS 调度同集群）；为「需要真机 IP」的请求分配节点并转发；与设备调度中心共享节点在线状态。
- **计费与风控**：按流量或按请求数计「虾粮」；隧道网关侧可限速、黑名单，防止单节点被滥用。

## 3. 应对「拔网线」的终极防御：任务补偿与状态机 (Task State Machine)

### 缺失环节

下沉市场网络极不稳定。若「点兵虾」把重要发帖任务派给某台网吧电脑，该电脑中途死机或断网，任务可能永久丢失。

### 补齐方案

- **位置**：在 **任务队列与打包器（如 RabbitMQ）** 下方引入 **死信队列 (Dead Letter Queue, DLQ)**，并在 **WSS 调度中心** 内实现 **任务状态机**。
- **状态机**：任务生命周期至少包含：`Pending → Assigned → Running → (Completed | Failed | Reclaimed)`。
  - **Assigned**：已下发给某节点（通过 WSS）。
  - **Running**：节点已 Ack 并开始执行（可带心跳）。
  - **Reclaimed**：调度中心在「毫秒级」内检测到节点断连或超时无心跳，将任务收回并重新入队，状态回到可派发。
- **DLQ**：多次重派仍失败（如重试 3 次后仍无健康节点可接）的任务进入 DLQ，供人工或告警处理，不丢单。
- **感知**：WSS 侧对每个已派发任务维护「预期心跳」；节点断连或心跳超时即触发回收逻辑，并在 1 秒内将任务重新派发给同城/同省备用节点（若架构支持地域亲和）。

### 效果

系统做到「毫秒级感知」节点失效；任务自动回收并秒级重派，MCN 侧无感知的「自愈」，不因单点断网而丢任务。

### 实现要点（建议）

- **调度中心**：为每个任务维护 `assignedNodeId`、`lastHeartbeatAt`、`retryCount`；定时扫描超时任务并执行 Reclaim → 重新入队。
- **消息队列**：主队列 + DLQ；Reclaim 时重新 publish 到主队列并增加 `retry_count`；超过阈值则 publish 到 DLQ。
- **边端**：任务执行期间按约定频率上报心跳（可复用现有 WSS 或独立心跳 topic），便于调度中心判断「运行中」与「失联」。

## 4. 防止被薅羊毛：本地硬件探针与反作弊 (Telemetry & Anti-Cheat)

### 缺失环节

用「虾粮（Token）」吸引小商家挂机时，可能出现虚拟机多开、脚本伪造在线时长等白嫖算力积分的行为。

### 补齐方案

- **位置**：在 **边缘节点客户端** 内埋入 **硬件特征探针 (Hardware Fingerprinting)**；在 **全局风控合规防火墙** 侧增加 **反作弊规则引擎**。
- **探针内容（示例）**：
  - 硬件：CPU 型号/核心数、内存大小、真实网卡 MAC（需权限）、磁盘序列号等；
  - 行为：真实鼠标移动轨迹（时间序列）、键盘输入节奏、屏幕分辨率与多屏信息；
  - 环境：是否虚拟机/沙箱、是否常见云主机特征、传感器数据（如 CPU 温度，若可获得）。
- **上报**：探针数据在节点注册或周期性上报时发给云端；云端做哈希/脱敏后与历史指纹对比，识别同一机器多开、机房批量注册等。
- **防火墙动作**：一旦规则引擎判定为「虚假节点」或「作弊」，在全局风控合规防火墙将该节点封杀（踢出调度池、不再派单、不结算虾粮），并可选告警与审计日志。

### 效果

池子里的 IP 尽量保持「野生原生态设备」，降低被平台识别为机房/脚本流量的风险，同时保护虾粮体系不被批量薅羊毛。

### 实现要点（建议）

- **客户端**：探针 SDK 与 RPA/执行器同机部署；采集需遵守隐私与最小必要原则；数据可先本地聚合再上报，减少频率与体量。
- **云端**：指纹库 + 规则引擎（规则可配置：如同一指纹多节点、虚拟机特征命中、行为序列过于规则等）；与「设备调度中心」和「虾粮算力账本」联动，实现封杀与停发积分。

## 5. 数据洪流的泄洪口：实时 OLAP 分析引擎

### 缺失环节

图右下角 PostgreSQL 存「虾粮账本」没问题；但若每天十万节点并发回传海量运行日志、线索数据、点赞转化率，传统关系型库会被冲垮，且不适合做复杂聚合与秒级大屏。

### 补齐方案

- **位置**：在数据层引入 **实时 OLAP 引擎**（如 **ClickHouse** 或 **Elasticsearch**），与现有 PostgreSQL/Redis 分工。
- **分工建议**：
  - **PostgreSQL / Redis**：账本、订单、用户、设备注册、任务元数据等强一致、事务型数据；
  - **ClickHouse / ES**：日志流、线索流水、行为事件、转化指标等**写入量大、读多写少、聚合查询多**的数据；写入可采用消息队列（如 Kafka）再批量入湖，避免直冲 PG。
- **SuperHarbor 报表**：大盘、战役效果、线索统计、节点健康汇总等，**只读**从 OLAP 查询；写操作（创建战役、派单）仍走现有 API 与 PG。

### 效果

MCN 在「超级海港」后台看数据报表时，支持**亿级数据毫秒级**刷新，大屏与导出丝滑不卡顿。

### 实现要点（建议）

- **写入链路**：边端/网关 → Kafka（或类似）→ Flink/消费任务 → ClickHouse 或 ES；对账与幂等由业务设计保证。
- **查询层**：SuperHarbor 后端封装 OLAP 查询接口（按时间、战役、节点、平台等维度聚合）；前端不变，仍调现有或新增 REST/GraphQL。
- **选型**：ClickHouse 更适合日志与指标型聚合；ES 更适合全文与复杂过滤；可按数据类型拆成多库或混合使用。

## 与现有架构图的对应关系（简要）

- **OTA**：与「WSS 集群与设备调度中心」、「任务队列与打包器」并列或内嵌；任务包内带脚本版本与下载地址，边端「RPA 拟人化执行器」以空壳 + 动态脚本方式运行。
- **P2P 隧道**：边缘物理节点多一个「Tunnel Client」；云端多一个「隧道网关」，与触须虾等爬虫/调用方对接，流量经边端真机 IP 出口。
- **任务状态机 + DLQ**：在「任务队列与打包器」与「WSS 调度中心」内实现；DLQ 作为队列层组件，状态机在调度服务内维护。
- **硬件探针与反作弊**：边端集成探针；「全局风控合规防火墙」扩展反作弊规则与封杀动作；可与「认证服务」「虾粮算力账本」联动。
- **OLAP**：在「虾粮算力账本 (PostgreSQL/Redis)」与「对象存储」之外，新增 ClickHouse/ES 及写入链路；SuperHarbor 只读报表走 OLAP。

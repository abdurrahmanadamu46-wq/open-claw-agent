## 一、项目定位

**Dragon Senate（龙虾池）** 是面向中国本地商家的 AI 增长操作系统。

核心产品形态：**1 个 Commander（元老院总脑）+ 9 只执行龙虾**，通过 SaaS + 代理商模式，帮助餐饮、美业、教育、汽车服务等本地商家完成内容生产、线索获取、客户跟进的全链路 AI 自动化。

### 龙虾的本质定义（红线级）

**龙虾不是 Agent，是有专业人格的角色扮演执行者。**

| 维度 | 传统 Agent | 我们的龙虾 |
|---|---|---|
| 定位 | 自主决策的独立智能体 | 有专业人格和知识体系的角色扮演执行者 |
| 驱动方式 | 自主感知-规划-行动循环 | 由 Commander 编排，接收任务包，按角色专长执行 |
| 服务模式 | 单任务独占 | 同时服务多个客户/租户，按优先级并发处理 |
| 身份 | 无人格，纯功能体 | 有名字、性格、专业领域、成长历程的"虾员工" |
| 约束风格 | 外部强约束（token budget、step limit） | 软约束（步数+时间预警，优雅退出，不强制截断） |

### 主要客群

**中国本地实体商家**，覆盖 13 个行业大类、73 个子行业：

| 酒店民宿 | 商务酒店、精品民宿、城市民宿 |
| 建筑行业 | 门楼 |

### 客户痛点

1. **内容生产成本高**：请代运营 ¥3,000-¥8,000/月/账号，效果不稳定
2. **缺乏行业专业知识**：通用 AI 工具不懂行业规则和平台规范
3. **线索跟进断层**：从内容到留资到成交，中间环节无人衔接
4. **多账号管理混乱**：抖音 + 小红书 + 视频号 + 公众号，人工跟不上

### 为什么我们有差异化优势

- **行业知识包**：73 个子行业 × 9 只龙虾 × 4 类知识文件 = 2,628 个行业专属 JSON，每只龙虾执行任务时自动注入行业规则、触发钩子、评分标准和金样例，通用 AI 工具做不到这个深度
- **全链路闭环**：从信号发现（radar）→ 策略规划（strategist）→ 内容生产（inkwriter/visualizer）→ 分发调度（dispatcher）→ 互动承接（echoer）→ 线索抓取（catcher）→ 成交跟进（followup）→ ROI 归因（abacus）→ 反馈回 strategist/radar，**9 条连接全部存在，闭环成立**
- **云端养虾 + 边缘发布**：龙虾的大脑（记忆/策略/进化）完全在云端，但**实际的账号操作（发内容/回私信）由安装在客户设备上的轻量 Edge 客户端用本地真实 IP 执行**，平台看到的是客户自己的设备在操作，最大程度规避风控
- **越用越强**：每次任务执行写入 BattleLog → quality_score EMA 驱动角色进化 → 客户的龙虾比竞争对手的龙虾更懂本地市场

> 来源：`COST_ANALYSIS_AND_PRICING_V7.md`，更新日期 2026-04-02

### 核心定义

```
席位定义：1 席 = 1 个社交媒体账号（抖音/小红书/视频号/公众号）
```

### 底线设在 ¥1,980 的四个理由

1. **成本安全边际**：单席 AI 成本 ¥884，¥1,980 是其 2.24 倍，每席始终保持正边际
2. **保护渠道定价秩序**：底线 ¥1,980 确保代理向下游至少按 ¥2,980 售出，代理每席毛利 ≥ ¥1,000，有推广动力，不会恶性竞争
3. **品牌定位**：¥1,980/账号/月 ≈ ¥66/天，传统代运营单账号 ¥3,000-¥8,000/月，大批量时仍显"超值"但不廉价
4. **V6 教训**：V6 底线 ¥1,480，边际贡献仅 ¥596，需要 472 席才能覆盖固定成本，风险过高

## 四、商业化路径

### 第一阶段：种子期（0 → 50 席）

**核心目标**：拿下第一个付费客户，验证产品-市场匹配。

- 切入行业：美业（美容院、美发店）+ 餐饮（中餐馆、火锅店），这两个行业痛点最强、决策链最短、能快速看到 ROI
- 销售方式：创始人直接拜访，自签 1-4 席（¥4,800/席），不走代理
- 目标：第一个付费客户比招 10 个人更重要，哪怕只收 ¥1,000/月的内测价也要签
- 盈亏：50 席混合价 ≈ ¥3,500 均价，MRR ≈ ¥17.5 万，尚未达到盈亏平衡，需融资或自有资金支撑

### 第二阶段：代理扩张期（50 → 500 席）

**核心目标**：建立代理商体系，用杠杆放大销售覆盖。

- 优先在 3-5 个城市招募区域代理（50 席档，¥2,480/席采购）
- 代理商培训：产品演示 SOP + 行业话术包（结合知识库 73 个行业）
- 平台侧净利润：500 席时约 ¥62.7 万/月，净利润率 48%
- 代理侧年净利：区域代理 ¥19 万，省级代理 ¥98 万，对代理有强吸引力

### 第三阶段：规模化期（500 → 3,000 席）

**核心目标**：省级总代体系 + 行业垂直 SaaS 化。

- 每省一个总代（300 席底线价 ¥1,980）
- 行业定制化：基于 73 个知识包，推出行业版套餐（美业版、教育版、餐饮版）
- 净利润率稳定在 52-56%，MRR 目标 ¥200-¥594 万

## 九、移动端开发规划

> 决策日期：2026-04-03  
> 策略：iOS 优先，复用 React Native/Expo 技术栈，与云端 API 直连（无需中继层）

### 定位

**Dragon Senate 移动端** = 商家随身操作控制台 + 移动节点（让龙虾获得摄像头/GPS/推送能力）。

不是独立 App，是云端大脑的手持延伸：商家老板在手机上启动任务、看结果、处理龙虾问的问题、批准内容发布。

---

### 技术选型

| 维度 | 选择 | 理由 |
|---|---|---|
| 框架 | **React Native + Expo SDK 52+** | 与前端 Next.js 共享 TypeScript 类型和 API client；Expo EAS Build 一键出包 |
| 状态管理 | Zustand（跟随 web 端） | 已有 web store 可抽取共享 hook |
| 导航 | Expo Router（文件路由）| 与 Next.js 路由习惯统一 |
| 网络 | TanStack Query（React Query）| 已在 web 端使用，直接迁移 |
| 连接模式 | **直连云端 API**（HTTPS + WebSocket）| 我们有公有服务器，不需要 Cloudflare Relay 中继层 |
| 推送 | Expo Push Notifications | 龙虾完成任务、线索入库时推送 |
| 平台优先级 | **iOS 先行 → Android 跟进** | 目标客群（本地商家老板）iOS 占比更高 |

### 移动节点能力（区别于桌面端的核心亮点）

商家手机注册为 Dragon Senate 边缘节点后，龙虾可调用：

| 能力 | 用途 |
|---|---|
| 摄像头 | 现场拍摄 → 自动生成门店素材，提交 visualizer |
| GPS 定位 | radar 识别本地热点活动，精准关联地理位置 |
| 系统推送 | 龙虾问题需人工决策时实时打断，不漏线索 |
| 相册访问 | 批量导入商家历史素材，进 visualizer 处理流 |

移动节点注册复用现有 `edge_auth.py` 注册协议，node_id = `mobile_{tenant_id}_{device_fingerprint}`。

### QR 码配对接入流程（借鉴 clawket 模式）

```
1. 运营控制台（Web）→ 设置页面生成临时 accessCode（6 位，5 分钟有效）
2. 手机 App → 扫码，发送 POST /api/mobile/pair { accessCode, device_info }
3. 后端验证 → 返回 clientToken（JWT，30 天有效）
4. App 持久化 clientToken，后续所有请求携带 Authorization: Bearer
5. 同时在 EdgeNode 表注册该设备，状态 STARTING → ONLINE（首次心跳后）
```

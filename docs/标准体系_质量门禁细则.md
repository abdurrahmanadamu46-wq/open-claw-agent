# ClawCommerce 质量门禁细则

> 总负责人要求：代码质量门可自动执行、可复查，与《PR 与合并规范》一致，确保商业化成品无低级缺陷。

---

## 1. 前端（Web 仓）

| 工具 | 配置要求 | 门禁标准 |
|------|----------|----------|
| **ESLint** | 使用 TypeScript 推荐规则 + React hooks；禁止 `no-console` 或仅 allow warn/error 且注释说明 | 0 error；warning 需在 PR 中说明 |
| **Prettier** | 与 ESLint 兼容（eslint-config-prettier）；单引号、尾逗号、printWidth 统一 | CI 中 `prettier --check .` 或 format 后无 diff |
| **TypeScript** | `strict: true`；禁止隐式 any（或仅 @ts-expect-error 并注释） | `tsc --noEmit` 通过 |
| **构建** | `next build` 成功 | 失败则阻塞合并 |
| **测试** | Playwright E2E（上线前）；单元测（若有） | 覆盖率基线由团队约定（建议关键路径 90%+） |
| **密钥/日志** | ESLint 规则禁止 `process.env.*` 在客户端硬编码密钥；禁止 `console.log` 提交（可 `console.warn/error` 且需注释） | 违反即 fail |

**建议**：`package.json` 中 `lint` 含 eslint+prettier；`lint:fix` 可自动修复；CI 执行 `lint` + `build` + `test`。

---

## 2. 后端（NestJS 仓）

| 工具 | 配置要求 | 门禁标准 |
|------|----------|----------|
| **ESLint** | TypeScript + Nest 推荐规则 | 0 error |
| **Prettier** | 与前端风格统一（便于全栈） | CI check 通过 |
| **TypeScript** | strict | `tsc --noEmit` 通过 |
| **测试** | Jest + Supertest；覆盖率 ≥85%（见手册） | 覆盖率不降；新增逻辑需对应用例 |
| **构建** | `npm run build` | 失败则阻塞 |
| **安全** | 无敏感信息写日志；依赖 `npm audit` 无 high/critical（或已建档豁免） | 可选：CI 中 `npm audit --audit-level=high` |

---

## 3. Agent（本仓）

| 工具 | 配置要求 | 门禁标准 |
|------|----------|----------|
| **ESLint** | TypeScript 严格 | 0 error |
| **Prettier** | 统一风格 | CI check 通过 |
| **TypeScript** | strict | `tsc --noEmit` 通过 |
| **测试** | Jest；与平台相关部分可 mock 或标记为集成测 | 核心路径覆盖；无 skip 未说明 |
| **构建** | `npm run build` | 失败则阻塞 |
| **密钥/日志** | 禁止提交 `.env`；禁止 console.log（或仅允许 logger） | lint 或 review 检查 |

---

## 4. 全仓通用

- **依赖**：禁止提交 `package-lock.json` 外未经审批的 lock 变更；可选 CI 中 `npm ci` 保证可复现。
- **大文件**：禁止提交二进制/大文件（可 .gitignore + CI 检查）；镜像与构建产物不进入代码库。
- **合并前**：main 分支保护规则建议：必须通过 CI、必须 Review（或指定人 bypass）、禁止 force push。

---

## 5. 门禁未通过时

- 开发在 48 小时内修复或说明豁免理由（总负责人/PM 同意后可暂时合并，但需开 Issue 跟踪补测）。
- 每周五交付自查表须填写「本周门禁通过率」与未通过原因。

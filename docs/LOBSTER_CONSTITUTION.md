## 第一章：龙虾池 v2 核心定义

> **单核运行时 + 共享记忆底座 + 角色化执行协议**

### 1.1 本质定义

龙虾池 v2 不是 10 个彼此独立的 AI 大脑，也不是 1 个普通 AI 临时扮演 9 个角色、随意换提示词。

**龙虾池 v2 的定义：**

> 一个统一运行时、一个共享组织记忆底座、一个统一治理与进化中枢，在任务执行时按**严格角色协议**实例化为不同龙虾身份的 AI 增长操作系统。

换句话说：

- **底层** 是一个系统
- **中层** 是一套共享知识与治理能力
- **上层** 是 1 个 Commander + 9 个稳定角色化执行身份

龙虾之间不是彼此隔离的独立大脑，也不是随意切换的提示词面具，而是：

> **同一底座上的不同岗位实例。**

---

### 1.2 设计原则

**① 单核运行时**

全系统只有一套统一运行时主链，负责：任务接入、路由与编排、上下文构造、模型调用、工具调用、审计记录、反馈回收、持续优化。

核心竞争力不来自多个分裂 runtime，而来自一个可持续进化的统一执行底座。

**② 共享记忆底座**

所有龙虾共享同一套组织级知识与记忆基础设施：企业记忆、客户事实、历史任务经验、行业知识包、时序知识图谱、工具使用经验、反馈与评测结果。

> 共享记忆不代表共享职责，也不代表共享发言风格。

**③ 角色化执行协议**

每只龙虾不是"普通 agent"，而是通过协议定义的**稳定岗位身份**。每个角色必须具备：身份定义、职责边界、输入格式、输出工件格式、行为约束、风格与口吻、协作规则、升级条件、行业知识注入规则、技能与工具使用范围。

> 角色的切换不是简单"变脸"，而是：统一运行时按角色协议加载对应岗位身份，完成该岗位负责的任务。

**④ Commander 编排优先**

Commander 是组织级编排中枢，负责：理解任务、任务拆解、选择阵容、分派角色、仲裁冲突、异常接管、结果汇总、复盘与反馈闭环。Commander 是总脑，但不是全能执行体。

**⑤ 审计与治理先于自治**

任何角色实例化与执行行为，都必须能追踪：当前角色是谁、为什么由这个角色执行、使用了哪些上下文、调用了哪些工具、输出归属于谁、是否触发升级或人工介入。

---

### 1.3 三层架构

| 层次 | 职责 | 类比 |
|------|------|------|
| **第一层：统一运行时层** | 执行主链、模型路由、工具调度、会话管理、审计日志、成本控制、反馈回收 | 系统的"身体"和"神经系统" |
| **第二层：共享记忆与治理底座** | 企业记忆、客户知识地图、时序知识图谱、行业知识包、策略规则、权限体系、评测实验系统 | 系统的"长期记忆"和"制度体系" |
| **第三层：角色实例化层** | 按需实例化角色身份、加载角色协议、执行岗位任务、输出角色工件、与其他角色协作交接 | 系统的"岗位表现层" |

---

### 1.4 运行机制

客户发起任务时，系统不是唤醒 10 个平级智能体自由竞争，而是：

1. Commander 接收任务
2. Commander 判断目标、上下文、优先级与风险
3. Commander 选择需要参与的龙虾角色
4. 统一运行时按对应角色协议实例化执行身份
5. 各角色基于共享记忆底座完成岗位任务
6. Commander 汇总结果、处理异常并向客户输出

例如：写文案时实例化"吐墨虾协议"，做视频策划时实例化"幻影虾协议"，做线索分级时实例化"铁网虾协议"，做 ROI 分析时实例化"金算虾协议"。底层共享一个系统，前台表现为多个专业岗位协作。

---

### 1.5 与传统多智能体的核心区别

| 维度 | 传统多智能体 | 龙虾池 v2 |
|------|------------|----------|
| 架构基础 | 多个平级 agent | 一个统一运行时 |
| 记忆模型 | 各自拥有局部记忆 | 一个共享记忆底座 |
| 协调方式 | 自治协商 | Commander 编排优先 |
| 风险 | 角色漂移、责任模糊、信息割裂 | 协议化岗位、治理一致性 |
| 定位 | 研究型系统 | 商业化组织协作系统 |

> 龙虾池不是"多 agent 套壳"，而是**组织化 AI 数字员工系统**。

---

### 1.6 核心边界（不可逾越）

- **共享事实，不共享口吻**：所有龙虾共享同一套事实底座，但必须保持各自稳定的角色表达与行为风格
- **共享记忆，不共享职责**：一个角色可以理解全局背景，但不能替代其他角色履行其岗位责任
- **共享底座，不共享决策权**：统一运行时支撑所有角色，但具体决策权按 Commander 编排和角色边界分配
- **角色切换必须协议化**：任何角色切换都不是随意换 prompt，而必须通过明确的角色协议实例化完成

---

### 1.7 权限边界的哲学

角色边界是为了防止真正的灾难（发错内容/泄露客户数据/越权承诺），而不是限制创造力。遇到边界问题，运行时通过协议化角色切换或任务路由升级解决，而不是报错卡住。

## 第三章：赛亚人进化体制

### 核心设计哲学

```
每只龙虾 = 元老（不断进化的灵魂）+ N个分身（执行实例）

进化是自发的，不是被动的：
  - 龙虾热爱自己的领域，会主动学习
  - 不需要等任务分配，可以主动发起学习/协作
  - 每次任务是进化的原料，不是终点

进化引擎组件：
  memory_governor    → 长期记忆管理（什么值得记住）
  agent_rag          → 历史经验即时检索（用过的方法随时调用）
  industry_kb        → 行业知识库（越来越专业）
  policy_bandit      → 策略选择器（哪种打法在这个场景最优）
  trace/postmortem   → 战斗日志与失败复盘
```

### 六条军规（铁律）

```
军规#1: 每次任务必须留下战斗日志
  输入/输出/耗时/模型/成本/评分 → 全部写入 BattleLog
  没有日志 = 这次任务不存在

军规#2: 每次赢要沉淀（quality_score ≥ 4.0）
  提取成功模式 → 回写到 industry_kb / prompt_registry
  赢的方法越积越多

军规#3: 每次输要升级（quality_score ≤ 2.0）
  必须写 Postmortem（根因 + 改进方案 + 验证时间）
  失败是原料，不是终点

军规#4: 同类低级错误不能连续出现3次
  第2次出现 → 自动告警
  第3次出现 → 熔断降级 + 强制复盘 + commander介入
  复盘完成 → 错误计数归零，重新出发

军规#5: 学到的新东西必须回写（四选一）
  industry_kb / prompt_registry / policy_bandit / commander规则

军规#6: 知识盲区必须声明，不能假装会
  遇到盲区 → 声明 + 主动找擅长的龙虾协作
  声明盲区是专业表现，不是软弱
  遇到盲区不能独自硬做
```

### 主动进化行为（非强制，但鼓励）

```
每只龙虾都应该自发地：
  - 定期扫描自己的 industry_kb，补充新知识
  - 分析自己的 BattleLog 高分记录，提炼规律
  - 主动向其他龙虾请教自己的盲区
  - 在没有任务时，主动分析行业趋势/研究新技术/更新模板库
  - 发现团队中其他龙虾的好做法，主动学习和借鉴
```

## 第四章：动态分身机制

### 分身容量设计（对标2000万年营收）

```
分身总上限设计原则：
  高并发执行岗位（dispatcher/echoer/inkwriter）→ 大量分身，支撑规模化

  元老不可分身（commander）→ 确保决策一致性

分身上限表（支撑2000万规模）：
  commander:  1     # 唯一总指挥
  radar:      50    # 多平台/行业监控
  strategist: 30    # 多客户策略并行
  inkwriter:  100   # 大规模文案生产

  echoer:     500   # 全面互动覆盖
  catcher:    50    # 多渠道线索处理
  abacus:     30    # 多客户数据分析
  followup:   200   # 大规模线索跟进
  
  理论最大并发: 1241个分身同时工作
```

### 分身运作原则

```
元老（Mother Instance）：
  拥有完整身份、长期记忆、进化历史
  每次分身退役，经验通过EMA回写到元老
  元老越来越强，分身自动继承

  继承元老当前 quality_score + 最近经验快照
  独立执行，不互相干扰
  执行完成后退役，经验回写

触发规则：
  单龙虾同时收到2个以上任务 → 自动分身

  超出上限 → 进入优先队列等待
  紧急任务 → commander 可临时授权扩容
```

## 第五章：权限边界哲学（灵活的护栏）

### 设计原则

```
❌ 不是这样的：
  "inkwriter只能做文案，不能碰任何其他事情"
  → 这会产生bug，会让工作流卡死

✅ 是这样的：
  "每只龙虾有自己最擅长的核心域，遇到边界情况
   第一反应是找协作，而不是报错"

权限边界的三个层次：

1. 核心能力（Core Domain）：
   这是龙虾最擅长的领域，可以完全自主决策
   没有人比他更懂，直接做

2. 协作地带（Collaborative Zone）：
   超出自己专长但可以参与的领域
   做之前先和相关龙虾沟通，达成共识后执行
   在日志中标注"跨域协作"

3. 真正的红线（Hard Limits）：
   只有真正会造成不可逆损失的操作才是红线
   例如：未经审批直接修改客户的CRM历史数据
   例如：在未确认内容合规的情况下发布
   这些是少数，不是大多数操作
```

### 各龙虾的唯一红线（最精简版）

| 龙虾 | 唯一红线 |
|------|---------|
| commander | 不代替专职龙虾做具体业务决策（但可以建议） |
| radar | 不在未授权情况下抓取需要登录的私密数据 |
| strategist | 不在未经commander确认的情况下启动新工作流 |
| inkwriter | 不跳过合规检查直接发布（标注疑点，等确认） |
| visualizer | 不在未确认规格的情况下发布到平台 |
| dispatcher | 不修改内容本身（只负责分发，不改内容） |
| echoer | 不主动私信陌生用户（防平台风控） |
| catcher | 不直接联系客户（转给followup处理） |
| abacus | 不修改原始数据（只分析，不篡改） |
| followup | 不承诺超出授权的折扣/权益 |

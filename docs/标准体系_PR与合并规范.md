# ClawCommerce PR 与合并规范

> 总负责人要求：单次 PR 必须 100% 满足本规范方可合并 main，确保商业化成品质量可追溯。

---

## 1. PR 必须包含的内容（缺一不可）

- [ ] **标题**：类型 + 模块 + 简述，例：`[Frontend] 线索页增加 TanStack Table 与导出`
- [ ] **描述**：本 PR 解决的问题、涉及的手册条款（如「对照手册 2 前端-线索管理页」）、是否含破坏性变更
- [ ] **关联**：关联的 Issue / 需求单（若有）
- [ ] **自检**：已本地通过 lint、format、test（见下文质量门禁）
- [ ] **文档与契约**（按变更范围勾选）：
  - [ ] 若改 API 契约：已更新或确认 Swagger/后端契约文档
  - [ ] 若改 UI：已更新或确认 Figma 文字原型 / textdesign 对应页面
  - [ ] 若改部署/环境：已更新 README 或 textinfra 部署说明
  - [ ] **变更日志**：在 CHANGELOG.md 或 PR 描述中写明「新增/变更/修复」条目，便于发布说明

---

## 2. 质量门禁（CI 必须全部通过）

| 门禁项 | 要求 | 负责 |
|--------|------|------|
| **ESLint** | 零 error；warning 需在 PR 中说明或消掉 | 全仓 |
| **Prettier** | 已格式化，无 diff（或 CI 自动 format check） | 全仓 |
| **TypeScript** | 严格模式，零 any 逃逸（或仅文档允许的例外） | 全仓 |
| **单元/集成测试** | 本次变更相关用例通过；覆盖率不低于当前基线（前端/后端/Agent 分别约定） | 各仓 |
| **密钥与日志** | 无 hardcode 密钥、无提交 console.log（可用 lint 规则拦截） | 全仓 |
| **构建** | 前端 build、后端 build、Agent build 成功（按仓） | 各仓 |

未通过门禁的 PR 不得合并；紧急 hotfix 须总负责人特批并补测。

---

## 3. Definition of Done（按交付物类型）

**前端功能（新页/新组件）**

- [ ] 仅通过 `src/services/` 请求数据，无裸 fetch/axios 直连
- [ ] 交互状态完整：loading、error、空状态、成功反馈
- [ ] 响应式与暗黑模式（若设计系统已要求）已适配
- [ ] 已补充或更新 textdesign 文字原型 / 截图说明
- [ ] 若有新接口依赖，已与后端/契约对齐并记录

**后端 API / 服务**

- [ ] 已更新或确认 Swagger（含请求/响应示例、错误码）
- [ ] 多租户/鉴权在该接口生效
- [ ] 有对应单测或集成测，覆盖率满足手册要求
- [ ] 无敏感信息写日志、无 SQL/命令注入风险

**Agent 能力（节点/二创/线索）**

- [ ] 与后端契约一致（execute/terminate/线索回传）
- [ ] 有单元或 E2E 覆盖；若涉及平台操作，有模拟或真实环境说明
- [ ] 日志可追踪、无密钥泄露；反检测/合规相关已自检

**DevOps / 配置**

- [ ] 部署文档或脚本已更新；环境变量已文档化
- [ ] 不影响现有流水线通过（或已同步更新 CI）

---

## 4. 合并与分支策略

- **main**：仅接受通过 CI 的 PR，且至少 1 人 Review（或总负责人/PM 指定豁免）。
- **发布**：从 main 打 tag（如 `v1.0.0`），发布说明基于 CHANGELOG 或 PR 汇总。
- **紧急修复**：可从 main 拉 hotfix 分支，合并回 main 后必须同步到发布分支并打补丁 tag。

---

## 5. 检查清单模板（可复制到 PR 描述）

```markdown
## PR 自检
- [ ] 标题与描述完整
- [ ] 本地 lint/format/test 通过
- [ ] 已更新文档/契约/CHANGELOG（按变更范围）
- [ ] 无 hardcode 密钥、无 console.log
- [ ] Definition of Done 已勾选（见标准体系_PR与合并规范）
```

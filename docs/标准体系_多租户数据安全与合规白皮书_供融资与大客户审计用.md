## 一、文档目的与适用范围

ClawCommerce 是面向商家的 B2B SaaS 产品，商家数据（线索、任务、账号信息）是核心资产。本白皮书说明我们在**租户数据隔离、敏感数据加密、操作审计、计费防击穿**四方面的设计与承诺，用于：

- **融资尽调**：向投资人展示技术底座满足企业级安全与合规要求。  
- **大客户审计**：响应客户安全问卷与合规审查，证明「A 商家看不到 B 商家数据」「隐私数据落盘即加密」「高权限操作可追溯」。  
- **合规自证**：对照《个人信息保护法》及行业实践，说明我们在技术层面如何落实「最小必要、安全存储、可审计」。

**适用范围**：当前及未来生产环境中的多租户数据与计费逻辑；具体实现以后端/Agent 代码与运维配置为准，本白皮书为抽象层面的承诺与机制说明。

## 二、100% 租户数据隔离机制（Tenant Isolation）

### 2.1 设计目标

杜绝「A 商家看到 B 商家线索或任务」的越权访问（即 BOLA：Broken Object Level Authorization）。从**身份识别 → 请求路由 → 数据访问**全链路强制租户隔离。

### 2.2 身份与租户识别

- **JWT 与 Tenant-ID**：用户登录后，后端签发 JWT。JWT 中携带当前用户的**租户标识（Tenant-ID）**，且不可被前端篡改（签名校验）。
- **全局请求上下文**：后端在收到请求后，从 JWT 中解析出 Tenant-ID，并写入**请求级上下文**（如 Node.js 的 AsyncLocalStorage）。后续所有业务逻辑在同一请求链中均可获取「当前租户」，无需逐层传参，避免遗漏。

### 2.3 数据访问强制隔离

- **数据库层**：所有与租户相关的表（线索、任务、账单等）均包含 `tenant_id` 字段。
- **查询强制约束**：我们承诺，**每一条面向租户数据的 SQL 查询都强制带有 `WHERE tenant_id = ?`**（参数来自当前请求上下文）。代码审查与规范要求禁止编写「跨租户」或「未带 tenant_id」的查询。
- **结果**：从物理/逻辑层面保证：即使发生误传参数或错误拼接，数据库也不会返回其他租户的数据；若未带 tenant_id，查询将无结果或报错，不会泄露他租户数据。

### 2.4 对外承诺（可写入合同或安全问卷）

- 「ClawCommerce 采用租户级数据隔离：所有租户相关数据访问均通过 JWT 中的 Tenant-ID 经请求上下文注入，且数据库查询强制带 tenant_id 条件，从机制上杜绝跨租户越权访问（BOLA）。」

## 三、核心资产的银行级加密（Data Encryption）

### 3.1 保护对象

- **线索库中的高敏感字段**：手机号、微信号等可直接联系到个人的信息（下称「核心资产」）。
- **合规依据**：《个人信息保护法》要求个人敏感信息在存储时采取加密等安全措施；行业实践采用字段级加密，确保「即使数据库被拖库，明文隐私数据也不泄露」。

### 3.2 加密方案

- **算法**：对上述核心资产采用 **AES-256-CBC** 进行**字段级加密**后落盘。列表中展示的为脱敏结果（如 138****5678），仅在商家主动「查看完整联系方式」且通过权限与审计后，由服务端解密返回明文。
- **密钥管理**：加密密钥与业务数据**隔离存储**（如独立密钥库或密钥管理服务）。密钥不随业务库一起备份或迁移；访问密钥的权限与访问业务数据的权限分离。这样即使业务数据库被拖库，攻击者也无法在无密钥的情况下还原明文，**满足《个人信息保护法》对安全存储与最小泄露风险的要求**。

### 3.3 对外承诺（可写入合同或安全问卷）

- 「ClawCommerce 对线索中的手机号、微信号等个人敏感信息采用 AES-256-CBC 字段级加密落盘；加密密钥与业务数据隔离存储。即使数据库被非法获取，未授权方也无法还原明文，符合《个人信息保护法》对个人信息的保护要求。」

## 四、零感知审计追踪（Audit Trail）

### 4.1 设计目标

对**高权限、高敏感操作**做到「谁、何时、做了什么」可追溯，满足 ToB 客户与监管对操作可审计的要求，且对正常业务用户**零感知**（无额外操作步骤）。

### 4.2 记录维度

- **谁（User）**：操作人身份（用户 ID / 租户 ID，视审计策略可包含角色）。  
- **何时（Time）**：操作发生时间（UTC + 时区或统一时间戳）。  
- **操作对象与类型**：例如「查看完整联系方式（Reveal Lead Contact）」——即调用了高权限接口、接触了明文敏感信息的行为。  

其他可纳入审计的操作包括但不限于：批量导出线索、修改 Webhook、修改订阅与计费相关设置等（按产品与合规需求扩展）。

### 4.3 存储与使用

- 审计日志写入独立存储（如独立表或审计库），与业务库分离，仅授权角色可查询；日志不可被业务用户或普通接口篡改。
- 用于：合规自查、安全事件排查、大客户/监管要求时的「操作证明」。

### 4.4 对外承诺（可写入合同或安全问卷）

- 「ClawCommerce 对高权限操作（如查看线索明文联系方式）记录审计日志，包含操作人、时间、操作类型；审计数据独立存储、不可篡改，可供合规与安全审计使用。」

## 五、防系统击穿与资损防线（Pessimistic Locking for Billing）

### 5.1 业务场景与风险

- 在高并发场景下，多路「线索回传」可能同时扣减同一商家的线索点数（余额）。若仅用普通更新，可能出现**超额扣减、余额变为负数**的资损与客诉。

### 5.2 技术措施

- **悲观锁（Pessimistic Locking）**：在扣减商家线索点数时，对当前租户的账户记录加**排他锁**（如数据库的 `SELECT ... FOR UPDATE`），在同一事务内完成「读余额 → 校验 → 扣减 → 写回」。这样，并发请求会串行化执行扣减逻辑，**从机制上保证余额不会被打成负数**，确保平台与商家的每一笔计费精确一致。
- **事务与回滚**：扣减与流水记录在同一事务中提交；任一步骤失败则整体回滚，不出现「扣了点数但没记流水」或「记了流水但没扣点数」的不一致状态。

### 5.3 对外承诺（可写入合同或安全问卷）

- 「ClawCommerce 在计费扣减环节采用悲观锁与事务控制，保证高并发线索回传时商家线索点数不会被扣成负数，计费准确、无资损风险。」

## 六、架构与运维层面的补充说明

- **内外网流量隔离**：Agent 与 Backend 部署在同一内部网络（如 Docker 网络 `claw_net`）。Agent 向 Backend 的线索回传、心跳等请求通过**内网地址**（如 `http://backend:3000`）访问，**不经过公网网关**。因此：
  - 内部高频心跳与线索回传**不占用公网限流配额**，也不暴露在公网上，降低被攻击面。
  - 公网仅暴露前端与对外 API，网关可对公网流量做限流、防 CC、HTTPS 等安全策略，实现**内外网物理级隔离**。

- **网关与 HTTPS**：对外入口采用反向代理/网关（如 Traefik），支持子域名隔离、全自动 SSL（如 Let's Encrypt）、安全 Header 注入与限流，满足行业通用安全基线。
